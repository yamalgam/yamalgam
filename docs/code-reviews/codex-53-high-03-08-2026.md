# yamalgam Code Review (Codex-53 High)

Date: 2026-03-08  
Reviewer: Codex (C/Rust security + architecture review)

## Scope

- Reviewed Rust code under `crates/`
- Reviewed test tooling under `tools/`
- Excluded vendored `libfyaml` implementation details as requested

## Review Method

- Static code review with targeted grep for panic/exec/limits paths
- Structural review of parser/scanner/composer/resource-limit wiring
- Test tooling behavior validation via:
  - `just test`
  - `cargo nextest run -p yamalgam-compare --test compliance --success-output=immediate ...`

## Executive Summary

The project is generally clean and disciplined (no `unsafe`, broad test coverage, clear crate boundaries).  
The highest-risk issues are in **compliance harness correctness** and **resource-limit enforcement gaps**:

1. Compliance tests currently **cannot fail on parser/scanner divergence**.
2. Compliance extraction only covers the **first `yaml:` case per test file**, skipping 14 suite cases.
3. Security-oriented limits (`max_alias_expansions`, `max_anchor_count`, `max_merge_depth`) are declared but not enforced.

These issues materially weaken confidence in correctness/security claims despite a large passing test count.

## Findings

### 1) High: Compliance harness does not fail on mismatches

- Evidence:
  - `crates/yamalgam-compare/tests/compliance.rs:80-115` logs `MISMATCH`/`EXPECTED`/`UNEXPECTED` paths but always returns `Ok(())`.
  - `crates/yamalgam-compare/tests/compliance.rs:135-175` does the same for event-level mismatches.
  - Runtime evidence:
    - `cargo nextest run -p yamalgam-compare --test compliance --success-output=immediate ...` produced:
      - `2 MISMATCH`
      - `15 EXPECTED`
      - `2 EVENT_EXPECTED`
    - Exit status remained success.
- Impact:
  - CI can be green while scanner/parser parity is broken.
  - Regressions become observability-only (logs), not enforcement.
- Recommendation:
  - Convert mismatch categories into assertions with an explicit allowlist mechanism if needed.
  - At minimum, fail on `MISMATCH`, `UNEXPECTED`, and `EVENT_UNEXPECTED`.

### 2) High: YAML test-suite ingestion drops multi-case files

- Evidence:
  - `crates/yamalgam-compare/src/c_baseline.rs:26-31` explicitly extracts only the first test case.
  - `extract_yaml_input` returns a single `Option<String>` (`c_baseline.rs:31-96`).
  - `compliance.rs` keys by filename stem only (`crates/yamalgam-compare/tests/compliance.rs:58-61`), so no per-case identity.
  - Example multi-case file: `vendor/yaml-test-suite/2G84.yaml:6,13,16,29`.
  - Count evidence:
    - `files=351`
    - `yaml cases=365`
    - `extra untested cases=14`
- Impact:
  - Coverage claims overstate actual exercised YAML Test Suite cases.
  - Edge-case regressions in secondary cases are invisible.
- Recommendation:
  - Parse all cases in each suite file and assign case-level IDs (`<FILE>#<index>`).
  - Generate/cache baselines per case, not per file.

### 3) High: Declared security limits are not enforced (`alias`/`anchor`/`merge`)

- Evidence:
  - Limits declared in `crates/yamalgam-core/src/loader.rs:159-163`:
    - `max_alias_expansions`
    - `max_anchor_count`
    - `max_merge_depth`
  - No usage found outside declarations/tests.
  - Parser/scanner currently enforce depth/key/scalar only:
    - `crates/yamalgam-parser/src/parser.rs:164`
    - `crates/yamalgam-scanner/src/scanner.rs:997,1316,1658,1830,2074`
  - Composer anchor/alias paths are unbounded:
    - anchor insertion: `crates/yamalgam-parser/src/compose.rs:197,218,262`
    - alias expansion via clone: `compose.rs:269-273`
    - merge collection with no depth accounting: `compose.rs:363-384`
- Impact:
  - DoS risk (memory/CPU) for adversarial anchor/alias/merge patterns.
  - Security posture advertised by `LoaderConfig` presets is incomplete.
- Recommendation:
  - Enforce these limits in composition/resolution logic where expansions happen.
  - Add dedicated stress tests for alias bombs, anchor floods, and deep merge chains.

### 4) Medium: Safe defaults are bypassed by convenience APIs

- Evidence:
  - `Parser::new` uses `ResourceLimits::none()` (`crates/yamalgam-parser/src/parser.rs:119`).
  - `Composer::from_str` calls `Parser::new` (`crates/yamalgam-parser/src/compose.rs:93`).
  - `from_str`/`from_str_single` route through `Composer::from_str` (`crates/yamalgam-parser/src/lib.rs:36-50`).
- Impact:
  - Common entry points default to unlimited parsing/composition behavior.
  - Callers must opt in manually and cannot pass config to `Composer::from_str` path.
- Recommendation:
  - Add `Composer::from_str_with_config` and route high-level APIs through moderate/strict defaults (or strongly document risk).

### 5) Medium: Invalid merge-key values are silently dropped

- Evidence:
  - Merge-key handling diverts `<<` to `collect_merge_pairs` (`crates/yamalgam-parser/src/compose.rs:240-245`).
  - Non-mapping merge values are ignored (`compose.rs:379-383`), and original key/value is not preserved.
- Impact:
  - Data loss and silent semantic drift (`<<: 1` disappears entirely).
  - Hard-to-debug behavior differences across YAML engines.
- Recommendation:
  - Either:
    - Treat invalid merge values as hard errors, or
    - Preserve original `<<` pair as normal mapping content.

### 6) Medium: C batch harness length parsing allows abusive allocation requests

- Evidence:
  - `tools/fyaml-tokenize/main.c:385` parses frame length with `atol`.
  - Length is cast to `size_t` then allocated (`main.c:386`).
  - No bounds check, no negative-value rejection.
- Impact:
  - Malformed batch protocol input can request huge allocations/OOM in tooling.
  - Can destabilize CI/local tooling runs.
- Recommendation:
  - Use `strtoull` with strict validation and an upper bound.
  - Reject negative/overflow/absurd lengths before allocation.

### 7) Low: Event comparison intentionally ignores tags

- Evidence:
  - `crates/yamalgam-compare/src/compare.rs:185-186,229-231` excludes `tag` from equality checks.
- Impact:
  - Tag-resolution regressions can pass as event matches.
- Recommendation:
  - Add an optional strict mode that compares tags (possibly after normalization).

## Recommended Next Actions (Priority Order)

1. Make compliance tests fail on real divergence (`MISMATCH`, `UNEXPECTED`, event equivalents).
2. Refactor YAML suite ingestion to case-level granularity.
3. Implement and test enforcement for alias/anchor/merge limits.
4. Add config-aware high-level parser/composer constructors with safe defaults.
