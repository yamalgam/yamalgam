## Where things stand

Fixed a parser bug where `DocumentEnd` tokens (`...`) in the directive prologue caused `parse_implicit_document_start` to error. Event compliance improved from 346/351 to 349/351. Both remaining TOKEN MISMATCH cases (M7A3 and BD7L) are now fully understood and documented.

## What changed

**Parser** (`crates/yamalgam-parser/src/parser.rs`):
- Added `seen_directive: bool` field to `Parser` — tracks whether a `%YAML` or `%TAG` directive has been seen in the current prologue
- Refactored `parse_implicit_document_start` from `match token {}` to `match kind {}` (extract kind before matching to avoid borrow-checker conflicts with `seen_directive`)
- New arm: `DocumentEnd` when `!seen_directive` — consumes bare `...` between documents and stays in `ImplicitDocumentStart`
- When `seen_directive` is true and `DocumentEnd` appears, falls through to the catch-all which correctly errors (directives require `---`)

**Tests** (`crates/yamalgam-parser/tests/parser.rs`):
- `bare_document_end_between_documents` — M7A3 input: two `...` with a comment between, followed by a block scalar
- `directive_then_document_end_is_error` — B63P input: `%YAML 1.2\n...\n` must be rejected

## Decisions made

- **M7A3 TOKEN MISMATCH stays** — the C scanner genuinely loses block scalar content (treats `%!PS-Adobe-2.0` as a directive, `# Not the first line` as a comment, emits empty scalar). Our scanner correctly produces the full block scalar value. This is a C bug, not ours.
- **BD7L TOKEN MISMATCH stays** — traced to libfyaml's `generated_block_map` flag (`fy-parse.c:3030-3033`). C gates `Key` emission on whether a `BlockMappingStart` exists at the current indent. Our scanner always emits `Key` when a simple key resolves. Both produce structurally invalid tokens for this `fail:true` input; both parsers reject. Not worth adding `generated_block_map` to match C on invalid input.

## Landmines

- **`seen_directive` must be cleared on both explicit `---` and implicit document start** — the flag is set when `VersionDirective` or `TagDirective` tokens are processed, cleared in the `DocumentStart` and catch-all content arms. Missing a reset would cause the next document's bare `...` to be incorrectly rejected.
- **B63P was accidentally "passing" before the fix** — our parser errored on the `DocumentEnd` token via the catch-all path (accidental correct rejection). The fix to handle `DocumentEnd` initially made B63P succeed incorrectly, which is why `seen_directive` gating was necessary.
- **`parse_implicit_document_start` now matches on `kind` (extracted), not `token` (borrowed)** — this was required to read `self.seen_directive` in the match guard without conflicting with `self.peek_token()`'s borrow. The `Some(_kind)` catch-all has an underscore prefix to suppress the unused variable warning.
