Each surface groups findings into a coherent concern area, not a category.yamalgam is a well-structured, zero-unsafe YAML parser with disciplined workspace organization and clean clippy output. The primary concerns are arithmetic edge cases in the serde deserializer that can panic on malformed event streams, stringly-typed errors that erase diagnostic information at each pipeline stage, and cross-crate performance costs from dynamic dispatch and missing inlining that benchmarks undercount due to LTO asymmetry between bench and release profiles.

The Untrusted Input Surface

The parser's defense-in-depth is compromised by unsigned arithmetic edge cases in the serde layer, where depth counters can underflow on malformed event streams — debug panics, release infinite loops.

skip_value depth counter underflows on leading end-events

Related: SeqAccess::drain and MapAccess::drain miss cross-type end-event guard Effort: trivial12-mo commits
significant

If the first event consumed is a SequenceEnd or MappingEnd (possible via from_tokens with a custom iterator, or a parser bug), depth decrements from 0. Debug builds panic; release builds wrap to u32::MAX and loop until the stream is exhausted — an infinite loop DoS. The pattern repeats in SeqAccess::drain and MapAccess::drain.

Use depth.checked_sub(1) and return Error::Unexpected on underflow.

SeqAccess::drain and MapAccess::drain miss cross-type end-event guard

Related: skip_value depth counter underflows on leading end-events Effort: trivial12-mo commits
significant

SeqAccess::drain has a depth==0 guard for SequenceEnd but lets MappingEnd fall through to the decrement. MapAccess::drain has the mirror bug. A stray cross-type end event underflows the counter.

Guard both end-event types at depth==0, or use checked_sub.

CST insert_whitespace_before indexes source without clamping span offset

Effort: trivial12-mo commits
moderate

Unlike source_text() which clamps with .min(self.source.len()), this path uses span.start.offset directly. If a parser event carries an out-of-bounds offset, the slice panics. In normal operation parser offsets are valid, but defensive library code should not panic on invariant violations it cannot control.

Clamp ws_end with .min(self.source.len()), matching source_text().

Scanner casts u32 column to i32 without overflow check

Effort: trivial12-mo commits
advisory

Mark.column is u32; the indent stack uses i32 with -1 as sentinel. Lines exceeding 2^31 characters wrap to negative, corrupting indent comparisons. With ResourceLimits::none() (the default constructor), input size is unbounded, making this theoretically reachable.

Use i32::try_from(column).unwrap_or(i32::MAX), or enforce max_input_bytes in the default ResourceLimits.

Two depth-underflow paths in the serde deserializer (skip_value and drain) are the most actionable findings in this audit. Both are small fixes with clear remediation. The CST indexing and column cast are lower-risk but follow the same pattern: trusting upstream invariants with arithmetic rather than with Result.

The Error Architecture Surface

Error types lose precision at each pipeline stage — scanner errors are stringly-typed, compose errors lack spans, and serde errors discard position information that users need for debugging.

ScanError is a stringly-typed bag with no variants

Enables: ComposeError variants carry only String, no structured data, Serde Error::Unexpected passes None span at 15+ call sites Effort: medium12-mo commits
significant

ScanError has a single String field. 51 error sites construct it with ad-hoc literals. Callers cannot programmatically distinguish error kinds without string matching. ParseError wraps ScanError, so the opacity propagates through the entire pipeline.

Add a ScanErrorKind enum alongside the message. Keep the message for Display but give callers variants for match arms.

Diagnostic lacks Display and std::error::Error impls

Effort: trivial12-mo commits
significant

Diagnostic is used as the E type in six public Result signatures across Input's API, but implements neither Display nor std::error::Error. Callers cannot use ? to propagate through anyhow::Error or Box<dyn Error>, and cannot format the error without manual field access.

Implement Display (formatting code + message) and Error for Diagnostic. Non-breaking addition.

ComposeError variants carry only String, no structured data

Enabled by: ScanError is a stringly-typed bag with no variants Effort: small12-mo commits
moderate

UnexpectedEvent and LimitExceeded use format!("{event:?}") strings. Callers cannot distinguish error sub-kinds without parsing the message. UndefinedAlias carries the name but no Span.

Add span fields and structured sub-data to variants.

Serde Error::Unexpected passes None span at 15+ call sites

Enabled by: ScanError is a stringly-typed bag with no variants Effort: medium12-mo commits
moderate

The Unexpected variant has an Option<Span> field but it is populated with None at 15+ call sites in de.rs. Custom(String) from serde::de::Error::custom never carries position info. Users debugging type mismatches get no indication of where in the YAML the error occurred.

Thread the most recently consumed event's span through to error construction sites. The Event already carries a span.

CST handle_error discards ParseError content

Effort: trivial12-mo commits
advisory

The ParseError is received as a function argument but dropped without being stored in CstNode::Error. Callers can see where an error occurred but not what went wrong.

Store the ParseError or its Display string in the error node.

The layered error types (ScanError, ParseError, ComposeError, serde::Error) have the right shape but the wrong payload. ParseError is well-designed with typed variants. ScanError, at the foundation of the pipeline, is a bare String — forcing every downstream consumer to either forward an opaque message or lose information. The serde layer has span fields but populates them with None at most call sites. Fixing ScanError is the highest-leverage error architecture change.

The Panic Discipline Surface

Library code relies on implicit invariants backed by expect/unwrap rather than Result, creating 40+ panic sites on paths reachable from the token stream.

Parser uses .expect("peeked") after peek_token — 40+ sites

Effort: small12-mo commits
moderate

Over 40 locations follow: peek_token() confirms Some, then next_token().expect("peeked") consumes it. The invariant is correct but depends on no interleaving calls between peek and next. Any future refactor that inserts a call between them causes a panic on user input.

Add a consume_peeked() helper returning Result that encapsulates peek-then-take. Eliminates 40+ panic sites.

Deserializer peek methods unwrap replay_buffer.front()

Effort: trivial12-mo commits
moderate

Four unwraps on replay_buffer.front() — each structurally safe (post push_front or confirmed via if-let) but the invariant is not local. A refactor removing the push_front turns these into panics.

Return the pushed reference directly, or use expect with descriptive message.

Composer::peek_event double-unwraps via action-integer pattern

Effort: small12-mo commits
advisory

Both unwrap() and unwrap_err() on one line. Structurally guaranteed by the preceding peek() match but the action-integer encoding obscures the invariant.

Restructure to avoid the action-integer pattern. Use explicit match on self.events.next() directly.

The invariants behind each unwrap are correct in the current code. The concern is not that they are wrong today but that they are fragile — each depends on a guarantee from another method with no compile-time enforcement. A consume_peeked() abstraction in the parser would eliminate the largest cluster. The serde replay buffer unwraps are structurally safe but could be made obviously safe.

The Hot Path Surface

The multi-crate parsing pipeline pays vtable dispatch, missing inlining, and per-event allocation costs that benchmarks undercount because bench profiles enable LTO but release profiles do not.

Parser uses Box<dyn Iterator> for token source

Related: Zero #[inline] annotations on hot Reader/Scanner functions, Release profile has no LTO — benchmarks misrepresent release perf Effort: medium12-mo commits
significant

Every call to self.tokens.next() goes through vtable dispatch. The concrete type is always Scanner in the normal case, but the indirection prevents inlining Scanner::next() into Parser's hot loop. Combined with no LTO in release builds, this is an unrecoverable optimization barrier.

Make Parser generic over the token iterator. Provide a type alias for the common Scanner case. The from_tokens() constructor can still accept any iterator — it just makes the Parser type carry the concrete type.

Zero #[inline] annotations on hot Reader/Scanner functions

Related: Parser uses Box<dyn Iterator> for token source, Release profile has no LTO — benchmarks misrepresent release perf Effort: trivial12-mo commits
significant

Without #[inline], cross-crate calls rely on LTO. The release profile has no LTO, meaning release builds get no inlining of these per-character functions. The bench profile uses lto="thin", hiding this cost from benchmarks.

Add #[inline] to peek(), peek_at(), advance(), mark(), is_eof(), slice(). These are tiny functions called thousands of times.

Reader::peek_at() is O(n) per call via chars().nth(n)

Effort: small12-mo commits
significant

chars().nth(n) iterates from current offset through n characters each call. The scanner calls peek_at() ~46 times, often with sequential small indices — is_document_start() does peek() + peek_at(1) + peek_at(2) + peek_at(3), each rescanning from offset. For ASCII-dominated YAML the constant factors are small, but for CJK/emoji content the rescanning is genuinely wasteful.

Cache the current char in Reader (set on advance). For small lookahead, a fixed-size char buffer eliminates rescanning. Or add a peek_byte() for ASCII-range checks that avoids char boundary overhead.

Release profile has no LTO — benchmarks misrepresent release perf

Enables: Zero #[inline] annotations on hot Reader/Scanner functions, Parser uses Box<dyn Iterator> for token source Effort: trivial12-mo commits
moderate

Bench profile adds lto="thin" but release does not. Cross-crate inlining happens in benchmarks but not in cargo install or release builds. Users get measurably worse performance than benchmarks report.

Add lto = "thin" to [profile.release].

Resolver::on_event allocates Vec per event on passthrough path

Effort: small12-mo commits
moderate

The NoopResolver (used in all default paths) allocates a Vec per event. For a 1000-event document, that is 1000 heap allocations producing no value. ADR-0007 originally specified SmallVec<[Event; 1]> for this reason.

Adopt SmallVec<[Event; 1]> or a None/One/Many enum that inlines the single-event case.

Serde deserializer clones Event on every peek-then-consume

Effort: small12-mo commits
moderate

The pattern peek_event()?.clone() + next_event() appears in deserialize_any (the most common serde dispatch path). Cloning an Event::Scalar with Cow::Owned value triggers a String allocation + memcpy. Every YAML value goes through this path.

Extract the discriminant info from the peek (just variant + style), drop the borrow, then call next_event() to get the owned event. Eliminates the clone entirely.

purge_stale_simple_keys() allocates Vec<u64> on every token fetch

Effort: small12-mo commits
moderate

Called from fetch_next_token() on every iteration. Vec::new() starts at zero capacity (no alloc until push), so the cost is zero when no keys are purged — which is the vast majority of calls. The finding is moderate because the allocation path is rare, but the code could avoid it entirely.

Use a bool flag during retain to track whether root tokens were purged, then iterate simple_keys again for the queue_position lookup. Or use SmallVec<[u64; 2]>.

Anchored events cloned twice during anchor buffering

Effort: small12-mo commits
moderate

When an anchored collection is encountered, all sub-events are collected into a Vec, then that Vec is cloned for the anchor registry, and the original is drained into the replay buffer. Every Event is cloned — including scalars with Cow::Owned strings. This doubles memory for every anchored subtree.

Move buffer into register_anchor, then replay from the anchors map. Or use Rc<[Event]> to share between anchor map and replay.

The three significant findings — dyn dispatch in the parser, missing inline annotations, and release-without-LTO — compound. Fix any one and the others matter less; fix all three and the entire cross-crate call chain becomes inlineable. The per-event Vec allocation in Resolver::on_event and the serde clone-on-peek pattern are independent costs that accumulate linearly with document size. These performance gaps explain why yamalgam may underperform on large inputs in release builds despite winning benchmarks.

The Supply Chain Surface

The dependency tree ships the very library this project aims to replace, and carries unused dependencies across multiple crates that inflate compile times without contributing functionality.

figment yaml feature pulls deprecated serde_yaml into every crate

Effort: trivial12-mo commits
significant

figment's yaml feature enables deprecated serde_yaml as a transitive dependency. serde_yaml pulls unsafe-libyaml (C code compiled via cc) and indexmap into every crate that depends on yamalgam-core — which is all of them. A YAML parser project shipping a deprecated YAML parser as a transitive dependency is an irony that also carries real supply chain weight.

Drop the yaml feature: figment = { features = ["toml", "json"] }. YAML config support can be restored via yamalgam's own parser.

Remove the unused entries from each Cargo.toml.

The API Design Surface

Public types are well-designed with correct lifetime propagation and zero-copy Cow usage, but miss standard trait implementations that idiomatic Rust consumers expect from map-like containers.

Mapping lacks IntoIterator impls

Effort: trivial12-mo commits
moderate

Callers must write for (k, v) in mapping.iter() instead of for (k, v) in &mapping. Consuming iteration is impossible without accessing the private entries field.

Implement IntoIterator for &Mapping, &mut Mapping, and Mapping. Non-breaking additions.

Value::get(&str) heap-allocates on every lookup

Effort: trivial12-mo commits
moderate

Every call constructs a Value::String via key.to_owned(), allocating on the heap, solely to linear-scan and compare. The allocation is dropped immediately after the scan.

Add Mapping::get_by_str that compares String variants directly against &str without allocating. Have Value::get delegate to it.

ResourceLimits is Clone but not Copy despite qualifying

Effort: trivial12-mo commits
advisory

ResourceLimits has 7 Option<usize> fields — all Copy. The type derives Clone but not Copy. The serde crate's from_str_with_limits takes it by value, requiring callers to .clone() for reuse. Inconsistent with from_str_with_config which takes &LoaderConfig by reference.

Derive Copy on ResourceLimits.

The API findings are all non-breaking additions. Mapping lacks IntoIterator, Value::get allocates per lookup, and ResourceLimits should derive Copy. These are polish items — the underlying design (streaming Iterator-based parser, Cow<'input, str>, layered crates) is sound. Lifetimes are well-applied throughout.
